# セキュリティレビュー

## 発動タイミング

認証/認可実装・ユーザー入力/ファイルアップロード処理・新APIエンドポイント作成・シークレット/認証情報扱い・支払い機能・機微データ保存/送信・サードパーティAPI連携

## セキュリティチェックリスト

### 1. シークレット管理
- [ ] APIキー・トークン・パスワードがハードコードされていない
- [ ] 全シークレットが環境変数に入っている
- [ ] `.env.local` が .gitignore に含まれている

### 2. 入力検証
- [ ] 全ユーザー入力をスキーマで検証
- [ ] ファイルアップロードを制限（サイズ・MIME・拡張子）
- [ ] ホワイトリスト検証を使用（ブラックリストでない）
- [ ] エラーメッセージで機微情報を漏らしていない

### 3. SQLインジェクション対策
- [ ] 全DBクエリでパラメータ化クエリまたはプリペアドステートメントを使用
- [ ] SQLに文字列連結を使っていない

### 4. 認証と認可
- [ ] トークンをhttpOnlyクッキーに保存（localStorageではない）
- [ ] 機微な操作の前に認可チェック
- [ ] RLS（行レベルセキュリティ）を有効化
- [ ] RBACを実装

### 5. XSS対策
- [ ] ユーザー提供HTMLをサニタイズ
- [ ] CSPヘッダーを設定
- [ ] フレームワーク標準XSS保護を使用

### 6. CSRF対策
- [ ] 状態変更操作にCSRFトークン
- [ ] クッキーに `SameSite=Strict` を設定

### 7. レート制限
- [ ] 全APIエンドポイントにレート制限
- [ ] 高負荷処理（検索・ファイルアップロード）には厳しい制限

### 8. 機微情報露出
- [ ] ログにパスワード・トークン・シークレットを出していない
- [ ] ユーザー向けエラーメッセージは汎用的

### 9. 依存関係セキュリティ
- [ ] 既知の脆弱性がない
- [ ] ロックファイルをコミット済み
- [ ] Dependabotを有効化

## セキュリティテスト

- 認証テスト: 未認証→401・無効トークン→401
- 認可テスト: 権限なし→403・自分のリソースのみアクセス
- 入力検証テスト: 無効入力→400・適切なエラーメッセージ
- レート制限テスト: 制限超過→429

詳細は `references/cloud-infrastructure-security.md` 参照。
