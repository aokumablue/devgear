# プロンプトチューニング

書き手が「明瞭」と思うほど別エージェントが詰まる。新規subagentに実行させ両面評価→最小修正を収束まで繰り返す。

## 発動条件

使う: スキル/コマンド/プロンプト新規作成・大幅改訂直後/エージェント挙動が期待外れで指示側の曖昧さが原因/頻繁使用スキル・自動化中核プロンプトの堅牢化

使わない: 使い捨てプロンプト（評価コスト > 効果）/主観的好みの反映が目的

## ワークフロー

### Step 0: description/body 整合チェック（静的・dispatch不要）

1. frontmatter `description` の trigger/用途を読む
2. body がカバーする範囲を読む
3. 乖離あり → iter 1 前に合わせる

**スキップ禁止**: subagent は description に合わせて body を「再解釈」→スキル要件未達でも精度が出る（false positive）

### Step 1: ベースライン準備

- **評価シナリオ** 2〜3種（中央値1 + edge 1〜2）: 対象プロンプトを実際に適用する現実場面
- **要件チェックリスト**: シナリオごとに成果物が満たすべき要件3〜7項目。精度% = 満たした項目数/全項目数。**事前固定・事後変更禁止**

### Step 2: バイアス排除実行

- Task tool で**新規 subagent を dispatch**（自己再読 NG）
- 複数シナリオ並列実行
- dispatch 不能時 → 「環境制約」節参照

### Step 3: subagent 実行

後述の**subagent 起動契約**に従ったプロンプトを渡す。

### Step 4: 両面評価

**実行者自己申告**（subagent レポートから抽出）: 不明瞭点 / 裁量補完 / テンプレ適用で詰まった箇所

**Trace 解釈**: 各不明瞭点に発生フェーズタグを付ける
- `Understanding` / `Planning` / `Execution` / `Formatting`
- Understanding の曖昧さ → Execution の連鎖失敗として表面化しやすい

**構造化振り返り**: 各不明瞭点を `Issue / Cause / General Fix Rule` の3点で返させる。`General Fix Rule` は失敗パターン台帳へのクラスレベル抽象化。

**指示側の計測**:
- 成功/失敗: `[critical]` タグ付き要件が**全て OK** のときのみ成功。1つでも NG or 部分的 → 失敗
- 精度: OK=1 / NG=0 / 部分的=0.5 で合算 → 全項目数で割る
- ステップ数: Task tool usage メタの `tool_uses`
- 所要時間: `duration_ms`
- 再試行回数: subagent 自己申告から抽出
- 失敗時: どの `[critical]` が落ちたかを"不明瞭点"節に1行添える

**要件チェックリスト制約**: `[critical]` を**最低1つ**含めること（0件だと成功判定が vacuous）。事後に付け外し禁止。

### Step 5: 差分適用

- 不明瞭点を潰す**最小修正**
- 1イテレーション1テーマ（関連する複数修正はOK、無関係は次回）
- **修正前に**: この修正が要件チェックリストのどの項目を満たすかを明示
- **失敗パターン台帳を先にスキャン**: `General Fix Rule` が既存エントリと一致するなら「既存修正がなぜ再発を防げなかったか」を先に問う

### Step 6: 再評価

新規 subagent で Step 2〜5 を回す（同一 agent 再利用禁止）

### Step 7: 収束判定

連続2イテレーションで以下を**全て**満たしたら停止（重要度高は3連続）:
- 新規不明瞭点: 0件
- 精度の前回比改善: +3pt 以下
- ステップ数の前回比変動: ±10% 以内
- duration の前回比変動: ±15% 以内
- **過適合チェック**: hold-out シナリオ1本追加評価。精度が直近平均から15pt 以上低下 → 過適合 → baseline 設計に戻る

## 評価軸

- **成功/失敗**: 意図した成果物を出したか（二値）→ 最低ライン
- **精度**: 要件を何%満たしたか
- **ステップ数**: ツール呼び出し数 → 指示の無駄遣い指標
- **所要時間**: `duration_ms` → 認知負荷の代替指標
- **再試行回数**: 指示曖昧さシグナル
- **不明瞭点**（自己申告）: 質的改善材料
- **裁量補完箇所**（自己申告）: 暗黙仕様の炙り出し

**重み付け**: 質的（不明瞭点・裁量補完）を主、量的（時間・ステップ数）を補助。時間短縮だけ追うとプロンプトが痩せすぎる。

### `tool_uses` の質的解釈

- 他シナリオ比**3〜5倍以上** → 自己完結性が低い
- 例: 全シナリオ 1〜3 なのに1シナリオだけ 15+ → recipe がスキル内になく references/ を横断探索
- 対処: iter 2 で「最小完成例 inline」や「いつ references を読むかの指針」を冒頭に追加

精度100%でも `tool_uses` の偏りがあれば iter 2 発動の根拠になる。

### 修正の波及パターン

- **保守的**: 1修正で複数軸狙ったが1軸しか動かなかった
- **上振れ**: 複数軸の判定文言を同時に満たした
- **ゼロ振れ**: 判定文言のどれにも届かなかった

差分適用前に「この修正が判定文言のどれを満たすか」を言語化して安定させる。

## subagent 起動契約

```
あなたは <対象プロンプト名> を白紙で読む実行者です。

## 対象プロンプト
<全文貼る or Read パスを指定>

## シナリオ
<状況設定1段落>

## 要件チェックリスト
1. [critical] <最低ライン項目>
2. <通常項目>
（判定規則は Step 4 に一元定義。[critical] は最低1つ必須。）

## タスク
1. 対象プロンプトに従ってシナリオを実行し、成果物を生成する。
2. 終了時に下記レポート構造で返答する。

## レポート構造
- 成果物: <生成物 or 実行結果サマリ>
- 要件達成: 各項目について OK / NG / 部分的（理由付き）
- **Trace**: 各フェーズに OK / stuck / skipped タグ。OK でないときは1行理由。
  - Understanding / Planning / Execution / Formatting
  - 4フェーズ全て OK なら `Trace: all OK` の1行で十分。
- **不明瞭点（構造化）**: 各項目3行:
  - Issue / Cause / General Fix Rule
- 裁量補完: 指示で決まっておらず自分の判断で埋めた箇所
- 再試行: 同じ判断をやり直した回数とその理由
```

## 環境制約

dispatch 不能な環境では**適用しない**。
- 代替1: ユーザーに別セッション起動を依頼
- 代替2: 「empirical evaluation skipped: dispatch unavailable」と明示報告
- **NG**: 自己再読で代替する

**構造審査モード**: 整合性・明瞭性チェックのみ。プロンプトに「今回は構造審査モード」と明記。empirical の代替ではなく補助（連続クリア判定には使えない）。

## 打ち切り基準

- **収束（停止）**: Step 7 の条件を満たしたとき
- **発散（設計を疑う）**: 3回以上イテレーションしても新規不明瞭点が減らない → 構造を書き直す
- **リソース打ち切り**: 重要度と改善コストが釣り合わなくなったら止める（80点で出す判断）

## 失敗パターン台帳

イテレーション横断で失敗モードを累積記録するリストを維持する。

```
- **Pattern名**: 短く記述的なハンドル
  - Example: <代表的な Issue 文言>
  - General Fix Rule: <クラスレベルのルール>
  - Seen in: iter N, iter M, ...
```

- Step 5 前に台帳をスキャンする
- `General Fix Rule` が既存エントリと一致するなら `Seen in` を更新し再発原因を調査
- 3回以上再発 → 「発散」基準にエスカレート
- 台帳は対象プロンプト単位で持つ

## バリアント探索（任意・停滞打破用）

収束基準に届かないとき、局所最適を疑って2バリアントを並列 dispatch:
- **Conservative**: 現在プロンプト + 次善の小修正
- **Exploratory**: 構造変更1つ（節並べ替え / 密な段落の分割 / 冗長節削除 / scaffolding 追加）

採用判定: 精度高 → 同率なら不明瞭点少 → 同率なら `tool_uses` 少

**注意**: subagent に「A vs B」を直接評価させない（位置バイアス・self-preference バイアス）。

コスト: dispatch 2倍。停滞疑いのときのみ使う。

## 提示フォーマット

```
## Iteration N

### 変更点（前回差分）
- <修正内容1行>
- Pattern applied: <台帳上のPattern名、または「(new)」>

### 実行結果（シナリオ別）
| シナリオ | 成功/失敗 | 精度 | steps | duration | retries | Weak phase |
|---|---|---|---|---|---|---|
| A | OK | 90% | 4 | 20s | 0 | — |
| B | NG | 60% | 9 | 41s | 2 | Execution |

### 構造化振り返り（今回新出）
- <シナリオB>: [critical] 項目N が NG — <落ちた理由1行>
  - Issue / Cause / General Fix Rule

### 裁量補完（今回新出）
- <シナリオB>: <補完内容>

### 台帳更新
- Added: <Pattern名>
- Re-seen: <Pattern名> — 既存修正が再発を防げなかった理由: <理由>

### 次の修正案
- <最小修正1行>

（収束判定: 連続X回クリア / 停止条件まであとY回）
```

## 禁忌事項

- 自分で読み直す → 客観視不可。新規 subagent を dispatch
- 1シナリオで充分 → 過適合。最低2、できれば3
- 不明瞭点ゼロが1回出たから終わり → 偶然あり。連続2回で確定
- 複数の不明瞭点を一気に潰す → 何が効いたか不明。1テーマずつ
- 関連する微修正も1件ずつ別 iter に → 逆方向の罠。関連する2〜3件は1iter でOK
- メトリクスが良いから質的フィードバックを無視 → 時間短縮は痩せすぎのサインにもなる
- 書き直した方が早い → 3回以上不明瞭点が減らないなら正解。それ以前は逃げ
- 同じ subagent を使い回す → 前回改善を学習。毎回新規に dispatch
