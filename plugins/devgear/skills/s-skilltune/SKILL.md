# プロンプトチューニング

書き手が「明瞭」と思うほど別エージェントが詰まる。新規subagentに実行させ両面評価→最小修正を収束まで繰り返す。

## 発動条件

使う: スキル/コマンド/プロンプト新規作成・大幅改訂直後/エージェント挙動が期待外れで原因を指示側の曖昧さに求めるとき/頻繁使用スキル・自動化中核プロンプトの堅牢化

使わない: 使い捨てプロンプト（評価コスト > 効果）/主観的好みの反映が目的（成功率改善でないとき）

## ワークフロー

### Step 0: description/body 整合チェック（静的・dispatch不要）

1. frontmatter `description` の trigger/用途を読む
2. body がカバーする範囲を読む
3. 乖離あり → iter 1前にdescriptionかbodyを合わせる

**スキップ禁止**: subagentはdescriptionに合わせてbodyを「再解釈」→スキル要件未達でも精度が出る（false positive）

### Step 1: ベースライン準備

- **評価シナリオ** 2〜3種（中央値1 + edge 1〜2）: 対象プロンプトを実際に適用する現実場面
- **要件チェックリスト**: シナリオごとに成果物が満たすべき要件3〜7項目。精度% = 満たした項目数/全項目数。**事前固定・事後変更禁止**

### Step 2: バイアス排除実行

- Task toolで**新規subagentをdispatch**（自己再読NG）
- 複数シナリオ並列実行 → 単一メッセージ内で複数Agent呼び出しを並べる
- dispatch不能時 → 「環境制約」節参照

### Step 3: subagent実行

後述の**subagent起動契約**に従ったプロンプトを渡す。実行者は実装・出力を生成し、最後に自己申告レポートを返す。

### Step 4: 両面評価

**実行者自己申告**（subagentレポートから抽出）: 不明瞭点 / 裁量補完 / テンプレ適用で詰まった箇所

**Trace解釈**: 各不明瞭点に発生フェーズタグを付ける
- `Understanding` / `Planning` / `Execution` / `Formatting`
- Understandingの曖昧さ → Executionの連鎖失敗として表面化しやすい

**構造化振り返り**: 各不明瞭点を`Issue / Cause / General Fix Rule`の3点で返させる。`General Fix Rule`は失敗パターン台帳へのクラスレベル抽象化

**指示側の計測**（判定規則は本節で一元定義）:
- 成功/失敗: `[critical]`タグ付き要件が**全てOK**のときのみ成功。1つでもNG or 部分的 → 失敗
- 精度: OK=満点 / NG=0 / 部分的=0.5 で合算 → 全項目数で割る
- ステップ数: Task tool usageメタの`tool_uses`
- 所要時間: Task tool usageメタの`duration_ms`
- 再試行回数: subagent自己申告レポートから抽出
- 失敗時: どの`[critical]`項目が落ちたかを"不明瞭点"節に1行添える

**要件チェックリスト制約**: `[critical]`タグ付き項目を**最低1つ**含めること（0件だと成功判定がvacuous）。事後に`[critical]`の付け外し禁止。

### Step 5: 差分適用

- 不明瞭点を潰す**最小修正**をプロンプトに入れる
- 1イテレーション1テーマ（関連する複数修正はOK、無関係は次回）
- **修正前に**: この修正が要件チェックリスト/判定文言のどの項目を満たすかを明示
- **失敗パターン台帳を先にスキャン**: `General Fix Rule`が既存エントリと一致するなら「既存修正がなぜ再発を防げなかったか」を先に問う

### Step 6: 再評価

新規subagentでStep 2〜5を回す（同一agent再利用禁止）

### Step 7: 収束判定

連続2イテレーションで以下を**全て**満たしたら停止（重要度高プロンプトは3連続）:
- 新規不明瞭点: 0件
- 精度の前回比改善: +3ポイント以下
- ステップ数の前回比変動: ±10%以内
- durationの前回比変動: ±15%以内
- **過適合チェック**: 収束判定時にhold-outシナリオ1本追加評価。精度が直近平均から15ポイント以上低下 → 過適合。baselineシナリオ設計に戻ってedgeを追加

## 評価軸

- **成功/失敗**: 意図した成果物を出したか（二値）→ 最低ライン
- **精度**: 要件を何%満たしたか → 部分成功の程度
- **ステップ数**: ツール呼び出し数 → 指示の無駄遣い指標
- **所要時間**: `duration_ms` → 認知負荷の代替指標
- **再試行回数**: 同じ判断のやり直し回数 → 指示曖昧さシグナル
- **不明瞭点**（自己申告）: 実行者の箇条書き列挙 → 質的改善材料
- **裁量補完箇所**（自己申告）: 指示で決まっていなかった判断 → 暗黙仕様の炙り出し

**重み付け**: 質的（不明瞭点・裁量補完）を主、量的（時間・ステップ数）を補助。時間短縮だけ追うとプロンプトが痩せすぎる。

### `tool_uses`の質的解釈

シナリオ間の相対値として使うと構造的欠陥が見える:
- 他シナリオ比**3〜5倍以上** → スキルの自己完結性が低い
- 例: 全シナリオ`tool_uses` 1〜3なのに1シナリオだけ15+ → そのシナリオ用recipeがスキル内になくreferences/を横断探索
- 対処: iter 2で「最小完成例inline」や「いつreferencesを読むかの指針」をSKILL.md冒頭に追加

精度100%でも`tool_uses`の偏りがあればiter 2発動の根拠になる。

### 修正の波及パターン

- **保守的に振れる**: 1修正で複数軸狙ったが1軸しか動かなかった
- **上振れ**: 構造的な情報の組合せが複数軸の判定文言を同時に満たした
- **ゼロ振れ**: 軸名から推測した修正が判定文言のどれにも届かなかった

差分適用前に「この修正が判定文言のどれを満たすか」を言語化して安定させる。

## subagent起動契約

```
あなたは <対象プロンプト名> を白紙で読む実行者です。

## 対象プロンプト
<対象プロンプトの本文を全文貼る or Readで読ませるパスを指定>

## シナリオ
<シナリオの状況設定1段落>

## 要件チェックリスト（成果物が満たすべき項目）
1. [critical] <最低ラインに含む項目>
2. <通常項目>
...
（判定規則は「Step 4: 両面評価 / 指示側の計測」節に一元定義。[critical]は最低1つ必須。）

## タスク
1. 対象プロンプトに従ってシナリオを実行し、成果物を生成する。
2. 終了時に下記レポート構造で返答する。

## レポート構造
- 成果物: <生成物 or 実行結果サマリ>
- 要件達成: 各項目について OK / NG / 部分的（理由付き）
- **Trace**（各フェーズについて OK / stuck / skipped タグを付け、OKでないときは1行理由を添える）:
  - Understanding（指示を読み、メンタルモデルを組み立てる）
  - Planning（アプローチ・順序を決める）
  - Execution（実際に作業する）
  - Formatting（成果物を期待される形に整える）
  - *折り畳み形式可*: 4フェーズすべてがOKのときは`Trace: all OK`の1行で十分。
- **不明瞭点（構造化）**: 各項目について3行:
  - Issue: <観測された現象>
  - Cause: <指示レベルで診断した原因>
  - General Fix Rule: <クラスレベルのルール>
- 裁量補完: 指示で決まっておらず自分の判断で埋めた箇所（箇条書き）
- 再試行: 同じ判断をやり直した回数とその理由
```

## 環境制約

新規subagent dispatch不能な環境では本スキルは**適用しない**。
- 代替1: ユーザーに別Claude Codeセッション起動を依頼
- 代替2: 評価を諦め「empirical evaluation skipped: dispatch unavailable」と明示報告
- **NG**: 自己再読で代替する

**構造審査モード**: スキル記述の整合性・明瞭性のみチェックしたい場合に使う。subagent依頼プロンプトに「今回は構造審査モード: 実行ではなくテキスト整合性チェック」と明記。empiricalの代替ではなく補助（連続クリア判定には使えない）。

## 打ち切り基準

- **収束（停止）**: Step 7の条件を満たしたとき
- **発散（設計を疑う）**: 3回以上イテレーションしても新規不明瞭点が減らない → 構造を書き直す
- **リソース打ち切り**: 重要度と改善コストが釣り合わなくなったら止める（80点で出す判断）

## 失敗パターン台帳

イテレーション横断で失敗モードを累積記録するリストを維持する。

エントリ形式:
```
- **Pattern名**: 短く記述的なハンドル
  - Example: <あるイテレーションの代表的なIssue文言>
  - General Fix Rule: <クラスレベルのルール>
  - Seen in: iter N, iter M, ...
```

ルール:
- Step 5で修正を生成する前に台帳をスキャンする
- `General Fix Rule`が既存エントリと一致するなら`Seen in`を更新し「既存修正がなぜ再発を防げなかったか」を調査
- 3回以上再発するパターンは構造的シグナル → 「発散」基準にエスカレート
- 台帳は対象プロンプト単位で持つ

## バリアント探索（任意・停滞打破用）

収束基準に届かないとき、局所最適を疑って2バリアントを並列dispatch:
- **Conservative**: 現在プロンプト + 次善の小修正
- **Exploratory**: 現在プロンプトに構造変更1つ（節並べ替え / 密な段落の分割 / 冗長節削除 / scaffolding追加）

採用判定: 精度高い方 → 同率なら不明瞭点少ない方 → さらに同率なら`tool_uses`少ない方

**注意**: subagentに「A vs B」を直接評価させない（位置バイアス・self-preferenceバイアス）。

コスト: dispatch2倍。停滞疑いのときのみ使う。

## 提示フォーマット

```
## Iteration N

### 変更点（前回差分）
- <修正内容1行>
- Pattern applied: <台帳上のPattern名、または「(new)」>

### 実行結果（シナリオ別）
| シナリオ | 成功/失敗 | 精度 | steps | duration | retries | Weak phase |
|---|---|---|---|---|---|---|
| A | OK | 90% | 4 | 20s | 0 | — |
| B | NG | 60% | 9 | 41s | 2 | Execution |

### 構造化振り返り（今回新出）
- <シナリオB>: [critical] 項目NがNG — <落ちた理由1行>
  - Issue / Cause / General Fix Rule

### 裁量補完（今回新出）
- <シナリオB>: <補完内容>

### 台帳更新
- Added: <Pattern名>
- Re-seen: <Pattern名> — 既存修正が再発を防げなかった理由: <理由>

```

## 禁忌事項

- 「自分で読み直せば同じ効果」→ 客観視不可。新規subagentをdispatch
- 「1シナリオで充分」→ 過適合する。最低2、できれば3
- 「不明瞭点ゼロが1回出たから終わり」→ 偶然あり。連続2回で確定判定
- 「複数の不明瞭点を一気に潰そう」→ 何が効いたか不明。1イテレーション1テーマ
- 「関連する微修正も1件ずつ別iterに」→ 逆方向の罠。"1テーマ"は意味単位。関連する2〜3件は1iterにまとめて良い
- 「メトリクスが良いから質的フィードバックは無視」→ 時間短縮は痩せすぎのサインにもなる。質的を主に
- 「書き直した方が早い」→ 3回以上不明瞭点が減らないなら正解。それ以前は逃げ
- 「同じsubagentを使い回そう」→ 前回改善を学習している。毎回新規にdispatch
