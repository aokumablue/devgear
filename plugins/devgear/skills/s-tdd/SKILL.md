# テスト駆動開発

## 発動タイミング

新機能実装・バグ修正（まず再現テスト）・コードリファクタリング・重要ビジネスロジック

## TDD サイクル

```
RED → GREEN → REFACTOR → REPEAT
```

**8段階の詳細手順:**

1. **SCAFFOLD** - インターフェイスを定義（関数シグネチャ・型・契約を明記）
2. **RED** - 失敗するテストを書く（まずテストが失敗することを確認）
3. テスト実行・失敗を確認する
4. **GREEN** - テストを通す最小限のコード実装
5. テスト実行・合格を確認する
6. **REFACTOR** - テストをグリーンに保ちながら実装を改善・最適化
7. テストを再実行してグリーンを確認する
8. カバレッジを確認（要件を満たしているか確認）

このサイクルを「テスト追加→実装→改善」で繰り返す。

## テスト種別

- ユニット: 独立した個別fn
- 統合: APIエンドポイント・DB操作

## エッジケース

Null/Undefined・空配列/文字列・無効型・境界値（最小/最大）・エラーパス・競合状態・大規模データ・特殊文字（Unicode・SQL文字）

## DO / DON'T ガイドライン

**DO（推奨）**
- 実装前にテストを書く - RED フェーズで失敗を確認してから GREEN へ
- 各変更後にテストを実行する - グリーン維持を確認
- テストで失敗確認する - 本当に失敗するテストか確認してから実装
- 最小限のコードを書く - テストを通すための最小実装のみ
- 振る舞いをテストする - 関数の出力・副作用・エラーハンドリングなど
- 外部依存（DB・API・ファイル）はモック - テストの独立性・再現性を確保

**DON'T（禁止）**
- テストより先に実装しない - 実装ファーストは TDD ではない
- 各変更後テスト未実行のまま進めない - グリーン状態を失うと後戻りが大きい
- 一度に大量のコードを書かない - 小さなサイクルで進捗を確認
- 実装詳細をテストしない - 内部ロジックではなく外部からの見え方をテスト
- 実装詳細の変更でテストが壊れないよう - テストの安定性を重視
- すべてをモックしない - 本当に外部依存が必要な箇所のみモック

## ベストプラクティス

- DO: 実装前にテスト書く / テストで失敗確認 / 最小コード書く / 振る舞いをテスト / 外部依存はモック
- DON'T: テストより先に実装 / 各変更後テスト未実行 / 一度に大量コード / 実装詳細をテスト / すべてをモック

## カバレッジ要件

**全体要件（CLAUDE.md に従う）**
- リポジトリ全体: 設定されたカバレッジ目標に達成する

**クリティカルコード（100%必須）**
- 金融計算・価格・決済ロジック
- 認証・認可・セッション管理
- セキュリティ（暗号化・検証・サニタイゼーション）
- コアビジネスロジック（重要な判定・計算・データ変換）

これらは 1 行でも未テストコードがあると本番環境でバグになる可能性があるため、カバレッジ 100% を必須とする。

## 永続メモリ

search: `test {機能ドメイン}` (file_pattern: *test*.py) / `bug fix error {関連キーワード}`
record: `{"event_type": "tdd", "content": "{テストサマリー: X tests, Y% coverage}"}`
参照: テストテンプレート / クリティカルコード検出 / カバレッジ傾向

## エージェント委譲

TDD サイクルの詳細な実装支援が必要な場合、`a-tdd` エージェントを明示起動してください。
エージェントは RED・GREEN・REFACTOR の各フェーズで実装の質をチェックし、テストカバレッジを確認します。
