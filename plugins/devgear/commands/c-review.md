# コードレビュー

## s-grillme 強制起動（必須）

開始直後に s-grillme を必ず起動し、完了まで他の処理に進まない。

## 永続メモリ

search: `review violation security` / `{変更ファイル名}` (file_pattern指定)
record: 必要時のみ、最終レビュー結果を1回だけ記録
参照: プロジェクト固有ルール / 頻出違反パターン。繰り返し違反は警告レベルを1段階上げ「繰り返し違反」とマーク。

## READ-ONLY 制約（絶対厳守）

このコマンドは**絶対にファイルを変更しない**。

- Edit / Write / MultiEdit ツールの使用を禁止する
- `git apply`, `sed -i`, `awk -i` 等のファイル書き換え Bash コマンドを禁止する
- 起動する a-review / a-secure も同様の制約に従う（tools リストに Write/Edit は含まれていないが明示的に禁止）
- 修正提案は「推奨修正」として**テキストで出力する**のみ。コード変更は禁止。

## スコープ確定（優先順）

1. 引数にパス指定がある場合:
   - ディレクトリパスの場合: そのディレクトリ配下の全ファイルを対象にする
   - ファイルパスの場合: そのファイルのみを対象にする
2. 引数がない場合: `git diff --name-only HEAD` の差分ファイルを対象にする

- 隣接ファイルへの自律的な拡散を禁止する

## レビュー実行（並列）

以下の2エージェントを**同時起動**し、両結果が揃ってからレポートを統合する:

**エージェントA: `devgear:a-review`**
品質・設計・保守性・BPレビュー。50行超fn/800行超ファイル/4階層超ネスト/エラーハンドリング不足/テスト不足/console.log/TODO-FIXME/a11y問題を確認。

**エージェントB: `devgear:a-secure`**
セキュリティ・脆弱性レビュー。ハードコード認証情報/SQLi/XSS/入力検証不足/パストラバーサルを確認。

## レポート内容

- 深刻度: CRITICAL, HIGH, MEDIUM, LOW
- ファイル位置と行番号
- 問題の説明
- 推奨修正
- **過去の類似違反**（memから取得した場合）

## CRITICAL または HIGH → commitをブロック

セキュリティ脆弱性あるコードは絶対承認しない!

## 後処理（指摘事項への対応）

レビュー完了後、指摘を分類して推奨手順を提示し、**ユーザー承認後にのみ** READ-ONLY 制約を解除して対応を実行する。承認なしの編集禁止。

### 分類

| 種別 | 判定 |
|---|---|
| バグ | 機能欠陥・誤動作・実害ある脆弱性 |
| リファクタ | 設計・可読性・保守性・性能・デッドコード |
| 両方 | バグ先行 → リファクタ |
| 対象外 | 仕様変更・大規模設計変更 |

### バグ対応手順

1. 再現テスト確立（失敗確認、再現不可なら不足情報を返して停止）
2. `s-grillme` で原因絞り込み → 最小修正方針決定
3. 根本原因を直接修正（正常系を壊さない）
4. 再現テストPASS + 回帰テスト追加 + 関連既存テスト再実行
5. 修正範囲のみ `a-review` / `a-secure` 再評価

### リファクタ対応手順

1. 対象を指摘ファイルに限定（隣接拡散禁止）
2. clean: `devgear:a-clean` でデッドコード削除、ファイル単位テスト → 失敗時 `git checkout -- <file>`
3. simplify: `devgear:a-simplify` で改善、グループ単位テスト → 失敗時リバート
4. perf: `devgear:a-perf` で性能修正、関連テスト → 失敗時リバート
5. review + secure 再評価: `devgear:a-review` / `devgear:a-secure` 並列実行、新規違反確認
6. final gate: テスト・linter 再実行、CRITICAL/HIGH 残存ならブロック

### 対象外対応手順

1. 指摘内容から仕様変更・設計変更の本質を抽出して要件を言い換え
2. `devgear:a-plan` / `devgear:a-arch`（評価モード）並列起動、依存関係とアーキ影響を特定
3. リスク・ブロッカー評価、複雑度（High/Medium/Low）見積もり
4. 変更ファイル一覧・非変更ファイル一覧・テストコマンドを含む段階的計画提示
5. ユーザー承認まで実装着手禁止（承認後は `c-bugfix` / `c-refactor` へ切り分け）

## 引数

$ARGUMENTS: `[ファイルパス or ディレクトリ]`（省略時: 変更差分）
