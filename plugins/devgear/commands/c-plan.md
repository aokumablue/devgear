# 実装計画

**planner** エージェントを呼び出し、コード記述前に実装計画を作成。

## 実施内容

1. **要件の言い換え** - 何を作るかを明確化
2. **過去の類似実装を検索** - memから関連する過去の作業を参照
3. **リスクの特定** - 潜在的な問題やブロッカーを洗い出す
4. **段階的な計画作成** - 実装をフェーズに分解
5. **確認待ち** - 進める前に必ずユーザー承認を得る

## 永続メモリ

search: `{要件のキーワード}` / `risk blocker error {関連キーワード}`
record: `{"event_type": "plan", "content": "{計画サマリー}"}`
参照: 類似機能実装履歴 / 過去のリスク・ブロッカー / 見積もり精度

## 使用タイミング

- 新機能開始時
- 大きなアーキテクチャ変更時
- 複雑なリファクタリング時
- 複数ファイル/コンポーネントに影響する時
- 要件が不明確または曖昧な時

## plannerエージェントの動作（並列）

以下の2エージェントを**同時起動**し、両結果をマージして最終計画を作成する:

**エージェントA: `devgear:a-plan`**
要件言い換え・手順分解・依存関係特定・複雑度見積もり

**エージェントB: `devgear:a-arch`（評価モード）**
アーキテクチャ影響調査・技術的リスク洗い出し・代替設計案提示

統合後:
1. 依頼を分析して要件を明確に言い換え（a-plan）
2. 具体的で実行可能な手順に分解（a-plan）
3. コンポーネント間の依存関係特定（a-plan）
4. リスクと潜在的なブロッカーを評価（a-arch評価で補強）
5. 複雑度（High / Medium / Low）見積もり
6. 計画提示→明示的な確認待ち

## CRITICAL

plannerエージェントは「はい」「進めて」などの肯定応答で計画確認するまで**コードを書かない**。

変更したい場合:
- "修正: [変更内容]"
- "別アプローチ: [代替案]"
- "フェーズ 2 を飛ばしてフェーズ 3 を先に進める"

## 連携

- 実装: `s-tdd` スキルが自動発火（または `a-tdd` エージェントを明示起動）してTDDを強制
- 完成後: `/c-review` でレビュー
