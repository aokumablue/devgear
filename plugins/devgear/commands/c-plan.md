# 実装計画

## s-grillme 強制起動（必須）

開始直後に s-grillme を必ず起動し、完了まで他の処理に進まない。

## 実施内容

1. **要件言い換え** — 何を作るかを明確化
2. **過去類似実装検索** — mem から関連作業を参照
3. **リスク特定** — 潜在的な問題・ブロッカーを洗い出す
4. **段階的計画作成** — 実装をフェーズに分解
5. **確認待ち** — ユーザー承認を得てから進む

## 永続メモリ

search: `{要件のキーワード}` / `risk blocker error {関連キーワード}`
record: `{"event_type": "plan", "content": "{計画サマリー}"}`

## 使用タイミング

- 新機能開始・大きなアーキテクチャ変更・複雑なリファクタリング
- **変更対象ファイルが5件以上のリファクタリング**
- 複数ファイル/コンポーネントに影響する変更・要件が曖昧な時

## plannerエージェントの動作（並列）

以下の2エージェントを**同時起動**し、両結果をマージして最終計画を作成:

- **`devgear:a-plan`**: 要件言い換え・手順分解・依存関係特定・複雑度見積もり
- **`devgear:a-arch`（評価モード）**: アーキテクチャ影響調査・技術的リスク洗い出し・代替設計案提示

統合後: 要件言い換え → 手順分解 → 依存関係特定 → リスク評価 → 複雑度見積もり（High/Medium/Low） → 計画提示→確認待ち

## CRITICAL

ユーザーの肯定応答（「はい」「進めて」等）まで**コードを書かない**。

計画の修正依頼があった場合は以下の形式で受け付ける:
- "修正: [変更内容]"
- "別アプローチ: [代替案]"
- "フェーズ 2 を飛ばしてフェーズ 3 を先に進める"

マルチファイルリファクタリング時は計画に含める:
1. 変更するファイル一覧（正確なパス）
2. 変更しないファイル一覧（隣接するが対象外）
3. テストコマンド

## 連携

- 実装: `s-tdd` 自動発火 または `a-tdd` 明示起動
- 完成後: `/c-review`
