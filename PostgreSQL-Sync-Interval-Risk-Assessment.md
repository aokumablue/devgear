# PostgreSQL同期間隔短縮リスク評価レポート
**現状：3時間間隔（既にハードコード化済み） → 検証課題：テスト・スクリプトの矛盾**

---

## 1. リスク列挙（技術的・運用的リスク）

### 1.1 性能リスク

**R1: データベース WAL 書き込み負荷 8倍増**
- 現状が3時間間隔であれば、24→3時間への短縮想定時、試行頻度は8倍に。実装上は既に3時間ハードコード化されているため、このリスクは**基本線**。
- WAL（Write-Ahead Log）量が8倍増加 → ディスク I/O 饽和、fsync コスト増大。
- **影響**: PostgreSQL チューニング不足時にディスク速度がボトルネック化。バッチサイズ500行でも累積効果は大。

**R2: 同期バッチの部分的・不完全同期リスク**
- 最大バッチサイズ500行制限により、1回の同期で全データを転送できない可能性。複数バッチをまたぐ場合、バッチ間での時系列整合性が崩れる。
- **影響**: チャンク・セッション・ADR等の複合的な関連データが別タイムスタンプで同期され、抽出時の時点の一貫性喪失。

**R3: リトライ間隔（5分）と同期間隔（3時間）の不調和**
- MIN_RETRY_INTERVAL = 5分。同期失敗時、5分後に再試行されるが、次の予定同期は3時間後。
- **影響**: 失敗後の復帰タイミング予測困難。スケジュール同期 vs 例外リトライの優先度不明確。

---

### 1.2 状態管理リスク

**R4: sync_state.json の同時アクセス・競合**
- ランタイム状態（last_synced_at, last_sync_attempt_at）をファイル永続化。複数スレッド/プロセスからの同時書き込み時にデータ破損の可能性。
- **影響**: 同期中の状態更新 vs 状態ファイル保存のレース条件。チェックポイント情報の欠落や矛盾。

**R5: 間隔短縮時の「前回同期時刻」初期化漏れ**
- 設定変更時に last_synced_at をリセットしない場合、初回同期までの待機時間が不確定。逆に初期化しすぎると同期漏れを引く。
- **影響**: 設定デプロイ後の同期スケジュールが予測不可能。ユーザー側で .devgear/sync_state.json を手動編集する必要が生じる。

---

### 1.3 テスト・一貫性リスク

**R6: テスト期待値の矛盾（24時間 vs 3時間 vs 168時間）**
- test_pg_setup_script.py は `interval_hours: 24` を期待（スクリプト出力）
- settings.py は `interval_hours: int = 3` をハードコード
- test_sync.py は `interval_hours = 168` でテスト実行
- **影響**: テストスイート実行時のカバレッジ不透明。実装と期待値が乖離したまま CI/CD で偽陽性。

**R7: embedded スクリプト（pg_setup.sh）の同期検証不足**
- pg_setup.sh が出力する初期化 SQL に `interval_hours: 3` がハードコード。設定ファイル化されていないため、環境別構成不可。
- **影響**: 本番環境でのみ24時間・ステージでは3時間といった環境別ポリシーが実装できない。

---

### 1.4 その他リスク

**R8: embedding 同期の複合的な遅延**
- chunks 同期 → embeddings 同期の2段階処理。同期間隔短縮時、chunk 未同期のまま embedding 処理が追い抜く可能性。
- **影響**: ベクトル検索の訓練データと実行時データのズレ。スキル実行時に古い embedding で検索される現象。

## 2. ブロッカー評価

### 2.1 B1: テスト期待値矛盾（Critical）
**根本原因:**
- pg_setup.sh スクリプトが静的に `"interval_hours": 3` 出力 → 正しい
- test_pg_setup_script.py が `"interval_hours": 24` を期待 → **テスト仕様が古い**
- 実装（settings.py）は `interval_hours: int = 3` → 実装が正しい

**影響:**
- CI/CD で test_pg_setup_script.py が失敗する
- デプロイメント阻止、リリース遅延

**解決案:**
```python
# test_pg_setup_script.py 修正
assert '"interval_hours": 3' in result.stdout  # 24 → 3 に変更
```
優先度: **P0（ブロッカー）**


### 2.2 B2: test_sync.py の 168時間テスト値が実装と乖離（High）
**根本原因:**
- test_sync.py は `interval_hours = 168` でテスト
- 実装は `interval_hours = 3` （デフォルト）
- テストと実装が別の間隔で動作・検証されている

**影響:**
- 短間隔（3時間）の同期ロジックが十分テストされない
- 本番環境で同期タイミングに関する回帰テスト不足
- should_sync() 関数の境界値テストが 168時間基準で設計されているため、3時間環境では異なる動作

**解決案:**
```python
# test_sync.py 追加テストケース
@pytest.mark.parametrize("interval_hours,delta_hours,expected", [
    (3, 2, False),      # 3時間間隔、2時間後 → 未達
    (3, 3, True),       # 3時間間隔、3時間後 → 達成
    (168, 167, False),  # 7日間隔、167時間後 → 未達
    (168, 169, True),   # 7日間隔、169時間後 → 達成
])
def test_should_sync_parametrized(interval_hours, delta_hours, expected):
    settings.sync.interval_hours = interval_hours
    settings.sync.last_synced_at = time.time() - (delta_hours * 3600)
    assert should_sync(settings) == expected
```
優先度: **P0（テストカバレッジ）**


### 2.3 B3: 設定値の階層化・管理戦略不明確（Medium）
**根本原因:**
- interval_hours がハードコード済みで、settings.json で上書き不可
- pg_setup.sh に静的な値が記述
- 本番/ステージ/開発環境で異なる値を使いたい場合、コード修正が必須

**影響:**
- 環境別ポリシーが実装できない
- インフラストラクチャ as Code 原則に違反
- デプロイ時にビルドアーティファクトの再生成が必要

**解決案（評価中）:**
- **案A**: settings.json に `sync.interval_hours` を追加（現在は enabled/postgres_url のみ）
- **案B**: 環境変数 `DEVGEAR_SYNC_INTERVAL_HOURS` でオーバーライド
- **案C**: 設定値をマイグレーション式（v1_default=24, v2_optimized=3）で段階デプロイ

優先度: **P1（今後の拡張性）**

## 4. パフォーマンス影響

### 4.1 定量分析

| 項目 | 24時間基準 | 3時間間隔時 | 係数 |
|------|-----------|-----------|------|
| 同期試行頻度 | 1回/日 | 8回/日 | **8倍** |
| 1回あたり平均行数 | ~4000行 | ~500行 | **1/8** |
| 日次 DB 書き込み量 | ~4000 row writes/日 | ~4000 row writes/日 | **同等** |
| WAL フラッシュ回数 | 1回/日 | 8回/日 | **8倍** |
| avg_query_latency | baseline | +15~25% | **予測** |
| connection pool 圧力 | 低 | 中 | **増加** |

### 4.2 スケーラビリティ限界計算

**前提:**
- SQLite: 1行平均 100バイト（メタデータ含む）
- PostgreSQL: 1行平均 150バイト（インデックス・キー含む）
- 両システム合算: 8回/日 × 500行 × 250バイト = **1MB/日**

**ボトルネック予測:**
- **接続数**: 1接続/同期 × 8回/日 = 8接続。pool size 20 なら問題無し ✓
- **ディスク I/O**: WAL = 8 fsync/日。SSD なら 10µs × 8 = 80µs/日。問題無し ✓
- **ネットワーク遅延**: 3時間間隔だと同期開始~完了の総時間 > 同期試行間隔。**積み残し風化リスク** ⚠

### 4.3 リスク評価図

```
同期試行頻度: 1回/日 ────────→ 8回/日
             (24h基準)     (3h 実装)
                |              |
                |              ├─ ✓ 日次書き込み量 同等
                |              ├─ ✓ スカラー接続負荷 OK
                |              └─ ⚠ WAL fsync 8倍
                |
CPU/ディスク/ネット <─ 3時間ごとの I/O burst
                ├─ バースト 500行 × 250byte = 125KB
                └─ 同期完了時間 > 次の同期予定時刻 → 積み残し
```

**結論**: 3時間間隔は技術的に実行可能。ただし **同期完了時間 > 3時間** になると、常に pending 状態に転移（理想的ではない）。

## 5. 代替案評価

### 案 A: 段階ロールアウト（Gradual）
**方針**: 24h → 12h → 6h → 3h の 4段階で漸進的に短縮

| 長所 | 短所 |
|------|------|
| 問題を段階的に検出 | デプロイメント回数増加（4回） |
| ロールバック容易 | テスト期間延長（1ヶ月+） |
| 本番負荷観察可能 | 管理オーバーヘッド大 |

**推奨**: 中規模チーム向け。初期段階 12h で1週間観察推奨。

---

### 案 B: 適応的同期間隔（Adaptive）
**方針**: pending 行数・DB 接続数に基づいて間隔を動的調整

```python
def adaptive_interval_hours(pending_count: int, conn_pool_usage: float) -> int:
    if pending_count > 5000 or conn_pool_usage > 0.8:
        return 1  # 危機時
    elif pending_count > 1000:
        return 3  # 通常
    else:
        return 6  # 低トラフィック時
```

| 長所 | 短所 |
|------|------|
| リソース効率的 | 実装複雑度 ↑（ロジック追加） |
| 本番自動チューニング | テスト困難（非決定性） |
| スパイク対応力 | 予測困難な挙動（監視必須） |

**推奨**: DevOps 中心チーム向け。Prometheus metrics 必須。

---

### 案 C: ハイブリッド同期（Hybrid）
**方針**: スケジュール同期（3h）+ イベント駆動同期（即座）

```python
# スケジュール: 3時間ごと
async def schedule_sync():
    while True:
        await asyncio.sleep(3 * 3600)
        sync_to_postgres(settings)

# イベント駆動: chunk 作成直後
def on_chunk_created(chunk):
    if settings.sync.enabled:
        asyncio.create_task(sync_to_postgres(settings))  # async 起動
```

| 長所 | 短所 |
|------|------|
| 即時性 + スケジュール性の両立 | 非同期実装必須 |
| UX 改善（すぐ検索可能） | リトライ戦略複雑化 |
| リソース平準化 | デッドロック・競合リスク増 |

**推奨**: 高可用性・SLA 要件ありのチーム向け。非同期 actor model 活用。

---

### 案 D: 設定ファイル化（Config-Driven）
**方針**: interval_hours を settings.json で上書き可能に

```json
{
  "mem": {
    "sync": {
      "enabled": true,
      "postgres_url": "...",
      "interval_hours": 3
    }
  }
}
```

修正コード:
```python
# settings.py SyncSettings
interval_hours: int = 3  # settings.json で上書き可
```

| 長所 | 短所 |
|------|------|
| 環境別ポリシー実現 | 設定ファイル スキーマ版管理必須 |
| ユーザ自己チューニング可能 | 検証ロジック追加（e.g. >= 1) |
| Infrastructure as Code 原則 | 本番環境でのミス設定リスク |

**推奨**: 全チーム向け（推奨度: ★★★）。settings.json schema 検証 + unit test で担保。

---

### 案 E: 負荷分散・マルチテナント同期（Load-Balanced）
**方針**: 複数クライアント/チームで同期時刻をずらす（群れ効果回避）

```python
# クライアント ID ハッシュで同期時刻をオフセット
offset_sec = hash(client_id) % (3 * 3600)
next_sync_time = last_synced_at + (3 * 3600) + offset_sec
```

| 長所 | 短所 |
|------|------|
| DB 負荷平準化 | クライアント ID 管理必須 |
| ハードウェア効率化 | 初期実装コスト高 |
| スケール性 ↑ | 監視・ダッシュボード複雑化 |

**推奨**: 大規模 SaaS 向け。初期段階では不要。
